## 1. Overview

go-daily supports **4 locales**:

| Code | Language             | File                       |
| ---- | -------------------- | -------------------------- |
| `zh` | Chinese (Simplified) | `content/messages/zh.json` |
| `en` | English              | `content/messages/en.json` |
| `ja` | Japanese             | `content/messages/ja.json` |
| `ko` | Korean               | `content/messages/ko.json` |

**Default locale**: `en` (English)  
**URL is source of truth**: All user-facing pages live under `/{locale}/...`, e.g. `/en/today`, `/zh/puzzles`.  
**Persistence**: The user's chosen locale is persisted via a cookie (`go-daily.locale`) for root-path `/` redirect negotiation. `localStorage` also caches it as a client fallback.

## 2. Directory Structure

```
content/messages/
  en.json   ← type authority (all locales must match this shape)
  zh.json
  ja.json
  ko.json

app/[locale]/          ← All pages under a locale segment
  layout.tsx           ← LocaleProvider (receives locale from URL)
  today/
  puzzles/
  ...

lib/
  localePath.ts        ← Locale negotiation, URL prefix/strip, Accept-Language parsing
  i18n.tsx             ← LocaleProvider + useLocale hook + localized() helper

components/
  LocalizedLink.tsx    ← next/link wrapper that auto-prefixes href with locale
```

## 3. How It Works

### 3.1 URL as Source of Truth

```
User visits /today
  → proxy.ts middleware detects no locale prefix
  → Negotiation priority: URL no override → cookie → Accept-Language → default en
  → 308 Permanent Redirect to /{locale}/today
```

Requests already carrying a locale prefix pass through; middleware injects the `x-locale` header:

```
User visits /ja/today
  → proxy.ts recognises `ja` as a valid locale
  → Writes x-locale: ja header
  → Also refreshes Supabase session
  → Passes through to downstream server components
```

### 3.2 Type Safety

The core design in `lib/i18n.tsx`:

```ts
import en from "@/content/messages/en.json";

// Messages type is inferred from en.json — no manual type definitions needed
type Messages = typeof en;

const DICTS: Record<Locale, Messages> = { zh, en, ja, ko };
```

**Key insight**: `type Messages = typeof en` makes `en.json` the single source of truth for the shape.  
When you add a new key to `en.json`, TypeScript immediately requires `zh.json`, `ja.json`, and `ko.json` to include the same key — compile-time error otherwise.

### 3.3 Context + Hook

```ts
type LocaleContextValue = {
  locale: Locale; // current locale code (from URL segment)
  setLocale: (l: Locale) => void; // switches locale (navigates to corresponding URL)
  t: Messages; // translated strings for the current locale
};
```

`LocaleProvider` wraps the app in `app/[locale]/layout.tsx`, receiving `initialLocale` from the URL segment. All child components can call `useLocale()` to access `locale`, `setLocale`, and `t`.

### 3.4 Locale Switch Flow

```
User clicks language switcher
  → setLocale("en")
    → setLocaleState("en")       // React state update, triggers re-render
    → localStorage.setItem(...)  // persisted for next visit
    → document.cookie = ...      // writes cookie for middleware
    → router.push(localePath("en", currentPath))  // navigates to new URL
```

**Note**: Switching locale navigates to a new URL (e.g. from `/zh/today` to `/en/today`), not just a cookie change. The URL is the single source of truth.

### 3.5 Eliminating SSR Locale Flash

`proxy.ts` handles locale negotiation at request time:

- If URL already has a locale prefix → use it directly
- If not → negotiate via cookie → Accept-Language → default en
- Negotiated result injected via `x-locale` header

`app/layout.tsx` (root layout) reads this header on the server:

```ts
const h = await headers();
const rawLocale = h.get("x-locale");
const locale: Locale = isLocale(rawLocale) ? rawLocale : DEFAULT_LOCALE;
```

This means `<html lang={locale}>` in the initial HTML is already rendered in the correct language — no hydration flash at all.

### 3.6 `LocalizedLink` — Auto Locale Prefix

```tsx
import { LocalizedLink } from "@/components/LocalizedLink";

// On /zh/today page:
<LocalizedLink href="/puzzles">Library</LocalizedLink>
// Renders as <a href="/zh/puzzles">

// External links, anchors, mailto auto-skip prefixing:
<LocalizedLink href="https://example.com" absolute>External</LocalizedLink>
```

`LocalizedLink` will:

- Auto-prefix relative paths with the current locale
- Skip external URLs, anchor links, `mailto:`
- Skip hrefs that are already absolute (e.g. `/en/puzzles`)

## 4. Using Translations in Components

### 4.1 Basic Usage (Client Components)

```tsx
"use client";
import { useLocale } from "@/lib/i18n";

export function MyComponent() {
  const { t, locale, setLocale } = useLocale();

  return (
    <div>
      <h1>{t.home.todaysProblem}</h1>
      <p>{t.result.correct}</p>
    </div>
  );
}
```

### 4.2 Interpolation (Parameterized Strings)

Some strings in `t.*` contain `{{placeholder}}` markers that need manual substitution:

```ts
// en.json: "tally": "{{solved}} solved · {{attempted}} attempted · {{unattempted}} todo"
const tally = t.puzzles.tally
  .replace("{{solved}}", String(stats.solved))
  .replace("{{attempted}}", String(stats.attempted))
  .replace("{{unattempted}}", String(stats.unattempted));
```

> There is no third-party i18n library (e.g. `next-intl`) — interpolation is manual. If interpolation needs grow significantly, consider adopting `next-intl` (see [FAQ](#8-faq)).

### 4.3 Server Components / RSC

Server components cannot use `useLocale()` (no hooks allowed).

**Page metadata (`generateMetadata`)** uses `getMessages(locale)` from `lib/metadata.ts`:

```ts
import { getMessages } from "@/lib/metadata";
import type { Locale } from "@/types";

export async function generateMetadata({ params }: { params: Promise<{ locale: Locale }> }) {
  const { locale } = await params;
  const t = getMessages(locale);
  return {
    title: t.metadata.today.title,
    description: t.metadata.today.description,
  };
}
```

The `metadata` section in each translation JSON (`content/messages/{zh,en,ja,ko}.json`) provides `title` and `description` for every page. Dynamic pages (e.g. `puzzles/[id]`) use `{{placeholder}}` template strings that are `.replace()`-d inside `generateMetadata`.

For other server-side scenarios, import the locale JSON directly:

```ts
import zh from "@/content/messages/zh.json";
const t = zh;
```

## 5. LocalizedText and the `localized()` Helper

### 5.1 LocalizedText

`Puzzle` fields `prompt` and `solutionNote` use the `LocalizedText` type:

```ts
type LocalizedText = Record<Locale, string>;
// i.e. { zh: string; en: string; ja: string; ko: string }
```

### 5.2 The `localized()` Function

Source: `lib/localized.ts`

```ts
import type { Locale, LocalizedText } from "@/types";
const FALLBACK_ORDER: Locale[] = ["en", "zh", "ja", "ko"];

export function localized(text: LocalizedText, locale: Locale): string {
  const v = text[locale];
  if (v) return v;
  // If the requested locale is missing, fall back in order
  for (const fb of FALLBACK_ORDER) {
    if (fb === locale) continue;
    const alt = text[fb];
    if (alt) return alt;
  }
  return "";
}
```

**Used for**: rendering puzzle text, building the AI coach system prompt, filtering in puzzle search.

**Never** write `puzzle.prompt[locale]` directly — it renders `"undefined"` when a locale field is empty.

**Correct pattern**:

```tsx
// ✅ Correct
import { localized } from "@/lib/i18n";
<p>{localized(puzzle.prompt, locale)}</p>

// ❌ Wrong (may render "undefined")
<p>{puzzle.prompt[locale]}</p>
```

## 6. Adding a New Translation Key

**Scenario**: you need a new UI string, e.g. `t.home.newFeature`.

1. **Add to `en.json` first** (the type authority):

   ```json
   "home": {
     "todaysProblem": "Today's Problem",
     "newFeature": "New Feature"   // ← add here
   }
   ```

2. **TypeScript will immediately error** on `zh.json`, `ja.json`, `ko.json`:

   ```
   Type '{ todaysProblem: string; }' is missing the following properties
   from type 'typeof en["home"]': newFeature
   ```

3. **Add the translation to the other three JSON files**:

   ```json
   // zh.json
   "newFeature": "新功能"
   // ja.json
   "newFeature": "新機能"
   // ko.json
   "newFeature": "새 기능"
   ```

4. Use `{t.home.newFeature}` in your component — IDE autocomplete included.

> **No dev-server restart needed** — Next.js hot-reload picks up JSON changes automatically.

## 7. Adding a New Locale

Example: adding French (`fr`).

### Step 1: Create the translation file

```bash
cp content/messages/en.json content/messages/fr.json
# Then translate all values in fr.json
```

### Step 2: Update `types/index.ts`

```ts
// Before
export type Locale = "zh" | "en" | "ja" | "ko";
// After
export type Locale = "zh" | "en" | "ja" | "ko" | "fr";
```

### Step 3: Update `lib/localePath.ts`

```ts
export const SUPPORTED_LOCALES: readonly Locale[] = ["zh", "en", "ja", "ko", "fr"];
```

### Step 4: Update `lib/i18n.tsx`

```ts
import fr from "@/content/messages/fr.json";

const DICTS: Record<Locale, Messages> = { zh, en, ja, ko, fr };
```

### Step 5: Update the fallback chain in `lib/i18n.tsx`

```ts
const FALLBACK_ORDER: Locale[] = ["en", "zh", "ja", "ko", "fr"];
```

### Step 6: Update the language-switcher UI

Add `fr` to the locale selector component with display text `"Français"`.

### Step 7: Update the validation script

`scripts/validatePuzzles.ts` has locale-checking logic — add `"fr"` to the relevant locale arrays there.

### Step 8: Update `app/[locale]/layout.tsx`

`generateStaticParams()` automatically generates static params from `SUPPORTED_LOCALES`. Ensure `lib/localePath.ts` is updated.

## 8. FAQ

### Q: Why not use `next-intl` or `react-i18next`?

At this scale (4 locales, ~100 string keys), a third-party library isn't needed. Context + TypeScript inference provides full type safety with zero extra dependencies.  
If interpolation templates (`{{placeholder}}`) grow substantially, or if you need pluralization rules or date formatting, `next-intl` is a natural fit — the migration cost is low because the APIs are similar.

### Q: I called `useLocale()` but the locale didn't change?

Check that your component is rendered inside `<LocaleProvider>` (already done in `app/[locale]/layout.tsx`).  
Calling `useLocale()` in an RSC throws an error — make sure the file has `"use client"` at the top.

### Q: What happens when a translation key is missing in some locale?

- `t.*` (UI strings): TypeScript enforces a consistent shape across all locale files at compile time — keys can't go missing under normal development.
- `puzzle.prompt` / `puzzle.solutionNote` (puzzle data): use the `localized()` function, which falls back to another locale automatically.

### Q: Will there be a locale-flash on hydration?

No. `proxy.ts` completes locale negotiation at request time, and the initial HTML's `<html lang={locale}>` is rendered in the correct language immediately. The URL is the source of truth — there's no cookie-vs-URL mismatch flash.

### Q: Why did the default locale change from zh to en?

go-daily targets an international audience. English as the default is better for SEO and first-time visitors. Chinese users can still use the site normally via `Accept-Language` or manual switching.
