## 1. Overview

go-daily supports **4 locales**:

| Code | Language             | File                       |
| ---- | -------------------- | -------------------------- |
| `zh` | Chinese (Simplified) | `content/messages/zh.json` |
| `en` | English              | `content/messages/en.json` |
| `ja` | Japanese             | `content/messages/ja.json` |
| `ko` | Korean               | `content/messages/ko.json` |

**Default locale**: `zh` (Chinese)  
**Persistence**: the user's chosen locale is stored in `localStorage["go-daily.locale"]` and restored on next visit.

## 3. How It Works

### 3.1 Type Safety

The core design in `lib/i18n.tsx`:

```ts
import en from "@/content/messages/en.json";

// Messages type is inferred from en.json — no manual type definitions needed
type Messages = typeof en;

const DICTS: Record<Locale, Messages> = { zh, en, ja, ko };
```

**Key insight**: `type Messages = typeof en` makes `en.json` the single source of truth for the shape.  
When you add a new key to `en.json`, TypeScript immediately requires `zh.json`, `ja.json`, and `ko.json` to include the same key — compile-time error otherwise.

### 3.2 Context + Hook

```ts
type LocaleContextValue = {
  locale: Locale; // current locale code
  setLocale: (l: Locale) => void; // switch locale
  t: Messages; // translated strings for the current locale
};
```

`LocaleProvider` wraps the entire app in `app/layout.tsx`, so all child components can call `useLocale()` to access `locale`, `setLocale`, and `t`.

### 3.3 Locale Switch Flow

```
User clicks language switcher
  → setLocale("en")
    → setLocaleState("en")       // React state update, triggers re-render
    → localStorage.setItem(...)  // persisted for next visit
```

## 4. Using Translations in Components

### 4.1 Basic Usage (Client Components)

```tsx
"use client";
import { useLocale } from "@/lib/i18n";

export function MyComponent() {
  const { t, locale, setLocale } = useLocale();

  return (
    <div>
      <h1>{t.home.todaysProblem}</h1>
      <p>{t.result.correct}</p>
    </div>
  );
}
```

### 4.2 Interpolation (Parameterized Strings)

Some strings in `t.*` contain `{{placeholder}}` markers that need manual substitution:

```ts
// en.json: "tally": "{{solved}} solved · {{attempted}} attempted · {{unattempted}} todo"
const tally = t.puzzles.tally
  .replace("{{solved}}", String(stats.solved))
  .replace("{{attempted}}", String(stats.attempted))
  .replace("{{unattempted}}", String(stats.unattempted));
```

> There is no third-party i18n library (e.g. `next-intl`) — interpolation is manual. If interpolation needs grow significantly, consider adopting `next-intl` (see [FAQ](#8-faq)).

### 4.3 Server Components / RSC

Server components cannot use `useLocale()` (no hooks allowed).  
Currently, all translated UI renders in client components (`"use client"`), so this isn't an issue.  
If server-side translation is needed in the future, import the locale JSON directly:

```ts
// server component (example only — not currently used)
import zh from "@/content/messages/zh.json";
const t = zh;
```

## 5. LocalizedText and the `localized()` Helper

### 5.1 LocalizedText

`Puzzle` fields `prompt` and `solutionNote` use the `LocalizedText` type:

```ts
type LocalizedText = Record<Locale, string>;
// i.e. { zh: string; en: string; ja: string; ko: string }
```

### 5.2 The `localized()` Function

Source: `lib/i18n.tsx`

```ts
import type { Locale, LocalizedText } from "@/types";
const FALLBACK_ORDER: Locale[] = ["en", "zh", "ja", "ko"];

export function localized(text: LocalizedText, locale: Locale): string {
  const v = text[locale];
  if (v) return v;
  // If the requested locale is missing, fall back in order
  for (const fb of FALLBACK_ORDER) {
    if (fb === locale) continue;
    const alt = text[fb];
    if (alt) return alt;
  }
  return "";
}
```

**Used for**: rendering puzzle text, building the AI coach system prompt, filtering in puzzle search.

**Never** write `puzzle.prompt[locale]` directly — it renders `"undefined"` when a locale field is empty.

**Correct pattern**:

```tsx
// ✅ Correct
import { localized } from "@/lib/i18n";
<p>{localized(puzzle.prompt, locale)}</p>

// ❌ Wrong (may render "undefined")
<p>{puzzle.prompt[locale]}</p>
```

## 6. Adding a New Translation Key

**Scenario**: you need a new UI string, e.g. `t.home.newFeature`.

1. **Add to `en.json` first** (the type authority):

   ```json
   "home": {
     "todaysProblem": "Today's Problem",
     "newFeature": "New Feature"   // ← add here
   }
   ```

2. **TypeScript will immediately error** on `zh.json`, `ja.json`, `ko.json`:

   ```
   Type '{ todaysProblem: string; }' is missing the following properties
   from type 'typeof en["home"]': newFeature
   ```

3. **Add the translation to the other three JSON files**:

   ```json
   // zh.json
   "newFeature": "新功能"
   // ja.json
   "newFeature": "新機能"
   // ko.json
   "newFeature": "새 기능"
   ```

4. Use `{t.home.newFeature}` in your component — IDE autocomplete included.

> **No dev-server restart needed** — Next.js hot-reload picks up JSON changes automatically.

## 7. Adding a New Locale

Example: adding French (`fr`).

### Step 1: Create the translation file

```bash
cp content/messages/en.json content/messages/fr.json
# Then translate all values in fr.json
```

### Step 2: Update `types/index.ts`

```ts
// Before
export type Locale = "zh" | "en" | "ja" | "ko";
// After
export type Locale = "zh" | "en" | "ja" | "ko" | "fr";
```

### Step 3: Update `lib/i18n.tsx`

```ts
import fr from "@/content/messages/fr.json";

const DICTS: Record<Locale, Messages> = { zh, en, ja, ko, fr };
```

### Step 4: Update the fallback chain in `lib/i18n.tsx`

```ts
const FALLBACK_ORDER: Locale[] = ["en", "zh", "ja", "ko", "fr"];
```

### Step 5: Update the language-switcher UI

Add `fr` to the locale selector component with display text `"Français"`.

### Step 6: Update the validation script

`scripts/validatePuzzles.ts` has locale-checking logic — add `"fr"` to the relevant locale arrays there.

## 8. FAQ

### Q: Why not use `next-intl` or `react-i18next`?

At this scale (4 locales, ~100 string keys), a third-party library isn't needed. Context + TypeScript inference provides full type safety with zero extra dependencies.  
If interpolation templates (`{{placeholder}}`) grow substantially, or if you need pluralization rules or date formatting, `next-intl` is a natural fit — the migration cost is low because the APIs are similar.

### Q: I called `useLocale()` but the locale didn't change?

Check that your component is rendered inside `<LocaleProvider>` (already done in `app/layout.tsx`).  
Calling `useLocale()` in an RSC throws an error — make sure the file has `"use client"` at the top.

### Q: What happens when a translation key is missing in some locale?

- `t.*` (UI strings): TypeScript enforces a consistent shape across all locale files at compile time — keys can't go missing under normal development.
- `puzzle.prompt` / `puzzle.solutionNote` (puzzle data): use the `localized()` function, which falls back to another locale automatically.

### Q: Will there be a locale-flash on hydration?

The server renders with the default locale (`zh`), then on client mount the saved locale is read from `localStorage` and triggers a re-render — this can cause a brief `zh → <user lang>` flash.  
To eliminate the flash, read the locale from a cookie server-side in `app/layout.tsx` (requires a server-side locale detection middleware).
